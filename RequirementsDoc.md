# 《迷你地下城指挥官》开发需求文档

## 1. 项目定位

《迷你地下城指挥官》是一款本地单机战术自动战斗 Roguelike 游戏。玩家组建三人小队，深入随机地下城，通过路线选择、技能构筑、装备词条和战斗策略击败敌人，最终挑战 Boss。

项目目标是完成一个可玩、稳定、视觉统一、可本地分发的完整小游戏作品。它需要重点体现战斗系统、角色状态机、敌人行为树、技能与 Buff 系统、数值配置、战斗复盘、UI 原型、美术素材生成、开发记录和最终打包交付。

项目需要与已有作品形成互补：

- 《像素挂机钓鱼》：挂机成长、存档、版本演进、素材接入和桌面端交付。
- 《夜市怪谈摊主》：AI Agent、LiveOps、运营数据、配置热更新和后台工具。
- 《迷你地下城指挥官》：战斗核心、角色 AI、状态机、技能系统、数值平衡和战斗表现。

## 2. 一句话介绍

玩家指挥一支三人冒险小队，在不断变化的地下城路线中自动战斗、收集遗物、强化技能，并通过战斗复盘理解每一场胜负背后的系统逻辑。

## 3. 项目亮点

项目需要突出以下亮点：

- 完整的自动战斗循环：从小队配置、路线选择、战斗执行到奖励强化，形成完整单局体验。
- 可解释的战斗系统：角色状态机、敌人行为树、目标选择、技能冷却和 Buff 触发都能被日志清楚记录。
- 有构筑深度的 Roguelike 体验：装备、遗物、技能强化和随机事件共同影响每一局的策略。
- 统一的地下城主题视觉：UI、角色、敌人、技能图标、背景和封面都需要围绕“地下城战术指挥”主题设计。
- 本地可部署和可分发：最终版需要打包成安装包或可执行文件，分享给他人后可以完整游玩。
- 完整项目记录：需求、架构、战斗系统、AI 行为、数值设计、素材生成、开发日志、版本记录和项目亮点介绍都需要文档化。

## 4. 技术形态

优先采用以下本地部署方案：

- 前端框架：React + TypeScript + Vite。
- 运行方式：开发阶段本地浏览器运行。
- 存档方式：localStorage。
- 视觉资源：Image-2 生成图片，必要时辅以 SVG / CSS。
- UI 设计：Pencil 原型设计。
- 桌面端：最终阶段使用 Tauri 或同等方案封装。

项目第一阶段不依赖后端、不依赖联网、不依赖真实大模型 API，保证本地稳定运行。

## 5. 核心玩法

### 5.1 基础循环

```text
选择小队 -> 进入地下城 -> 选择路线 -> 自动战斗 -> 获得奖励 -> 强化构筑 -> 挑战 Boss -> 查看战报
```

### 5.2 玩家目标

- 在有限生命值和资源下通关地下城。
- 根据敌人类型选择合适路线。
- 通过装备、技能和天赋形成构筑。
- 在战斗复盘中理解胜负原因。

### 5.3 单局时长

单局目标时长控制在 5 到 10 分钟，适合快速游玩、重复尝试和展示核心机制。后续内容扩展后，可以提供 10 到 20 分钟的完整游玩路线。

## 6. 主要模块需求

### 6.1 小队系统

首版提供 3 名默认角色：

- 守卫：前排坦克，擅长嘲讽、护盾和承伤。
- 游侠：单体输出，擅长暴击、连击和收割。
- 术士：后排法术，擅长群体伤害、燃烧和控制。

每个角色需要包含：

- 生命值。
- 攻击力。
- 防御力。
- 速度。
- 暴击率。
- 技能列表。
- 当前状态。
- Buff / Debuff 列表。

### 6.2 自动战斗系统

战斗采用自动执行，玩家主要在战前、路线中和战后做策略选择。

战斗系统需要支持：

- 回合 / 时间轴推进。
- 普通攻击。
- 技能冷却。
- 目标选择。
- 伤害计算。
- 治疗。
- 护盾。
- 暴击。
- 闪避或格挡。
- 角色死亡。
- 战斗胜负判定。

### 6.3 状态机

角色状态至少包括：

- Idle：等待行动。
- ChoosingTarget：选择目标。
- Casting：释放技能。
- Recovering：行动后恢复。
- Stunned：眩晕。
- Dead：死亡。

展示模式中需要展示当前角色状态变化，方便玩家理解战斗系统如何运转。

### 6.4 敌人行为树

敌人不能只是随机攻击，需要有可解释行为。

首版敌人行为类型：

- 低血量时优先自保。
- 优先攻击后排输出。
- 优先攻击血量最低目标。
- 当玩家有护盾时切换破盾技能。
- Boss 根据阶段切换技能策略。

展示模式中需要展示敌人的决策原因，例如：

```text
骸骨队长选择“破甲斩”：目标守卫拥有最高仇恨，且护盾值大于 20。
```

### 6.5 技能系统

技能数据需要配置化，便于扩展、平衡和复盘。

技能字段建议：

- id。
- name。
- owner。
- cooldown。
- targetRule。
- basePower。
- damageType。
- effects。
- description。

首版技能类型：

- 单体伤害。
- 群体伤害。
- 护盾。
- 治疗。
- 嘲讽。
- 中毒。
- 燃烧。
- 眩晕。
- 易伤。

### 6.6 Buff / Debuff 系统

Buff / Debuff 需要具备持续时间和触发时机。

触发时机包括：

- 回合开始。
- 行动前。
- 行动后。
- 受到攻击时。
- 造成伤害时。
- 回合结束。

首版效果：

- 护盾。
- 攻击提升。
- 防御提升。
- 易伤。
- 中毒。
- 燃烧。
- 眩晕。
- 嘲讽。

### 6.7 装备和构筑

战斗胜利后提供三选一奖励。

奖励类型：

- 武器。
- 护甲。
- 饰品。
- 技能强化。
- 被动天赋。

装备词条示例：

- 暴击率 +8%。
- 每次击杀后回复 10% 生命。
- 护盾值提高 20%。
- 燃烧伤害提高 30%。
- 战斗开始时获得 1 层加速。

### 6.8 地下城路线

地下城由多个节点组成，玩家每一层选择一个节点前进。

节点类型：

- 普通战斗。
- 精英战斗。
- 事件。
- 商店。
- 休息。
- Boss。

首版可以使用固定随机种子生成路线，保证调试和展示稳定。

### 6.9 事件系统

事件用于展示叙事和策略选择。

事件示例：

- 发现旧祭坛：献祭生命换取装备。
- 遇到迷路商人：购买随机遗物。
- 地下泉水：回复生命或获得临时 Buff。

事件选择需要进入日志，方便复盘。

### 6.10 战斗日志和复盘

项目必须有清晰的战斗日志，让玩家能理解自动战斗发生了什么。

日志需要记录：

- 每次行动。
- 技能释放。
- 目标选择原因。
- 伤害和治疗数值。
- Buff / Debuff 添加和移除。
- 击杀。
- 战斗结果。

复盘面板需要展示：

- 总伤害。
- 总承伤。
- 总治疗。
- 技能使用次数。
- 关键转折点。
- 胜负原因总结。

### 6.11 调参面板

项目需要内置一个开发用调参面板，便于平衡战斗和验证配置。

调参内容：

- 玩家角色基础属性。
- 敌人基础属性。
- 技能倍率。
- 敌人数量。
- Boss 阶段阈值。
- 掉落奖励权重。

调参面板只需要本地生效，不需要后端。最终分发版可以隐藏或放入开发入口。

### 6.12 展示模式

展示模式用于呈现游戏效果和系统设计，是项目的重要特色。

开启后展示：

- 当前战斗状态机。
- 敌人行为树决策。
- 技能冷却队列。
- Buff / Debuff 列表。
- 随机种子。
- 伤害公式。
- 最近 10 条关键事件。

目标是让玩家在观看战斗时，也能看到系统内部逻辑如何影响战斗结果。

展示模式不能做成复杂的大面板。默认只展示精简摘要，状态机详情、行为树详情、伤害公式和完整日志需要通过展开按钮或标签切换查看。

## 7. UI 设计需求

### 7.1 UI 复杂度限制

UI 需要尽量在一个电脑屏幕范围内完成主要展示，不追求复杂后台式布局。

要求：

- 核心游玩界面应尽量避免纵向长滚动。
- 主战斗画面、小队状态、敌人状态和主要操作需要在一个屏幕内可见。
- 复杂文本框、战斗日志、系统说明、数值公式和行为树详情需要做折叠处理。
- 可通过按钮展开 / 收起文本框。
- 可通过按钮或标签页切换不同信息面板。
- 默认状态只显示当前决策所需信息，详细信息由玩家主动展开。
- 移动端适配不是首要目标，优先保证常见笔记本和桌面显示器上的清晰体验。

### 7.2 设计工具

使用 Pencil 进行 UI 原型设计。

Pencil 路径：

```text
C:\Program Files\Pencil\Pencil.exe
```

需要产出：

- 主界面原型。
- 小队配置界面原型。
- 战斗界面原型。
- 路线选择界面原型。
- 战斗复盘界面原型。
- 调参面板原型。
- 展示模式界面原型。

UI 原型文件建议保存到：

```text
docs/ui/
```

### 7.3 视觉风格

所有 UI 和图片都必须切合游戏主题，形成统一的“地下城战术指挥”视觉风格。

建议方向：

- 整体氛围：地下城、烛火、石砖、古旧地图、战术棋盘、冒险者营地。
- UI 气质：清晰、硬朗、略带暗黑奇幻感，但不能牺牲可读性。
- 主色建议：炭黑、深铁灰、暗红、旧金、冷蓝法术光。
- 按钮和面板：像战术指挥桌、羊皮纸战报、金属铭牌或石刻面板。
- 角色与敌人：采用同一绘画风格，不混用明显不同画风。
- 技能图标：保持统一构图、统一边框和统一光效语言。
- 背景：需要能看出地下城层级变化，例如入口大厅、墓穴长廊、熔火祭坛、Boss 王座间。

要求：

- 信息层级清楚。
- 战斗区域突出。
- 角色状态易读。
- 技能冷却和 Buff 状态可见。
- 展示模式不能遮挡主要游戏内容。
- 复杂文本内容默认折叠，避免界面被大段说明占满。
- 避免过度装饰，优先保证可读性和操作效率。

### 8.2 首版素材清单

需要生成或制作：

- 3 名玩家角色立绘或头像。
- 6 到 8 个敌人头像或战斗单位图。
- 1 个 Boss 图。
- 3 到 4 张地下城背景。

- Buff / Debuff 状态图标。
- 项目封面图。

### 8.3 素材风格要求

所有素材必须围绕同一主题和同一画风生成。

建议统一提示方向：

```text
dark fantasy dungeon tactical RPG, candlelit stone chamber, heroic adventurer party, readable game asset, consistent painterly style, clean silhouette, no text
```

禁止出现：

- 与地下城主题无关的现代科幻元素。
- 明显卡通低龄化风格。
- 角色、敌人、图标之间画风割裂。
- 图片中包含不可控文字。
- 过暗导致无法识别主体的素材。

## 12. 开发阶段规划

### 阶段 1：项目骨架和可玩 MVP

目标：

- 建立 React + TypeScript + Vite 项目。
- 完成基础 UI。
- 实现 3 名角色和 3 种敌人。
- 实现自动战斗。
- 实现基础战斗日志。

验收标准：

- 可以打开本地页面。
- 可以开始一场战斗。
- 战斗能自动结束。
- 能看到行动日志和胜负结果。

### 阶段 2：战斗系统完整化

目标：

- 加入状态机。
- 加入敌人行为树。
- 加入技能冷却。
- 加入 Buff / Debuff。
- 加入伤害统计。

验收标准：

- 每个角色有清晰状态。
- 敌人行动有决策原因。
- Buff / Debuff 能正确生效和过期。
- 战斗复盘能展示关键数据。

### 阶段 3：Roguelike 构筑

目标：

- 加入地下城路线。
- 加入战斗奖励三选一。
- 加入装备和遗物。
- 加入随机事件。
- 加入 Boss 战。

验收标准：

- 可以完成一局 5 到 10 分钟的流程。
- 每局有构筑变化。
- Boss 战有阶段变化。

### 阶段 4：展示模式和调参面板

目标：

- 加入展示模式。
- 加入调参面板。
- 展示状态机、行为树、数值公式和随机种子。

验收标准：

- 可以直接看到系统内部逻辑。
- 调参后能立即影响战斗。
- 能用一套演示脚本讲清楚项目亮点。

### 阶段 5：美术、文档和交付

目标：

- 用 Pencil 完成 UI 原型。
- 用 Image-2 完成主要素材。
- 完成 README 和项目文档。
- 完成桌面端打包。
- 在 GitHub 上创建最终 Release。

验收标准：

- 项目视觉完整并且切合地下城主题。
- 文档完整。
- 本地运行稳定。
- 可执行文件或安装包可以分享给他人完整游玩。

## 13. 本地运行要求

开发阶段运行方式：

```bash
npm install
npm run dev
```

构建方式：

```bash
npm run build
```

桌面端：

```bash
npm run desktop:dev
npm run desktop:build
```

最终分发不要求接收者安装 Node.js、Rust、npm 或开发依赖。

## 15. 质量要求

项目需要满足：

- 没有明显控制台报错。
- 页面在桌面端和常见笔记本分辨率下可用。
- 战斗不会卡死。

- 随机逻辑可通过种子复现。
- 关键数值和配置集中管理。
- 视觉风格统一，UI 和图片都切合地下城战术主题。
- 主要界面尽量在一个电脑屏幕范围内展示，复杂文本信息需要折叠或切换显示。
- 最终分发包在无开发环境的电脑上可以正常运行。

## 16. 风险和应对

### 16.1 系统复杂度过高

应对：

- 先完成单场战斗 MVP。
- 再扩展路线、装备和 Boss。
- 所有系统都以可玩和可交付为目标，避免无限扩展。

### 16.2 美术资源耗时

应对：

- 第一版先用主题统一的占位图和简洁图标。
- 核心玩法稳定后再批量使用 Image-2 生成素材。
- 每批素材生成前先确定统一风格提示词。

### 16.3 数值平衡困难

应对：

- 保留调参面板。
- 记录默认配置。
- 使用固定种子和推荐构筑验证关卡节奏。

### 16.4 最终打包遗漏运行时依赖

应对：

- 最终版单独建立打包验收清单。
- 在干净环境或未安装开发依赖的环境中测试安装包 / 可执行文件。
- Release 中只上传可供游玩的分发物，不上传 node_modules、源码构建缓存或开发依赖。

### 16.5 UI 信息过载

应对：

- 优先设计一屏内可完成主要操作的布局。
- 长日志、复杂说明和系统详情默认折叠。
- 使用展开按钮、标签页和弹层承载低频信息。
- 每个界面只保留当前阶段最重要的操作和状态。

## 17. 额外建议

建议加入以下亮点功能：

- 固定演示种子：保证流程稳定。
- 一键展示按钮：直接进入准备好的战斗场景。
- 战斗回放：允许复看上一场战斗关键帧。
- 配置导出：导出当前角色、敌人和技能配置 JSON。
- 失败原因总结：例如“后排被优先击杀”“缺少驱散”“Boss 二阶段爆发过高”。
- README 首页放三张截图：战斗界面、展示模式、战斗复盘。

这些功能会让项目更像一个成熟的完整游戏作品，而不是普通小游戏 Demo。
